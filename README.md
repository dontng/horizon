# Horizon · 四战 160 天控制系统

> Horizon 不负责制造更多计划，而负责把有限精力稳定转化为考试产出。

## 北极星

- **目标院校**：华中师范大学
- **剩余时间**：160 天
- **当前主线**：先堵住睡眠、工作拖尾、恢复无边界和项目替代学习四个漏点，再逐步提高训练密度
- **唯一判断标准**：每周是否产生更多、质量更高、可以持续的考试训练

现在不缺目标，也不缺资料。真正缺的是一个能根据真实状态做取舍的控制系统：今天能做多少、只做哪一件、做到什么程度、什么时候必须停止。

## 已建立的事实基线

详细数据与推导已拆出，不再藏在对话或 README 中：

- [睡眠基线与修复计划](data/sleep/README.md)
- [工作基线与下班收口](data/work/README.md)
- [时间管理项目研究与 Horizon 取舍](research/time-management/README.md)

核心事实：

1. 最近 30 夜平均睡眠约 **5.34 小时**，中位入睡约 **03:12**，30 夜全部不足 7 小时。
2. 工作日通常约 **08:50 上班、20:05 下班**，通勤十几分钟；真正吞噬晚间精力的是工作拖尾和下班后无边界恢复。
3. GitHub、Codex、Claude Code 能制造较高构建产出，但不能计入考试训练。
4. 当前作息容量不足以支持“每天四科齐推”，必须采用容量调度和在制品限制。

## Horizon 的控制闭环

```text
真实状态输入
→ 判断今日容量
→ 选择唯一主任务
→ 设定完成线与时间上限
→ 执行并记录真实产出
→ 标记一个主要漏损
→ 周复盘只修最高频漏洞
```

Horizon 不替代具体学习项目。它只负责决定资源投向、边界和下一动作。

## 每日容量档位

| 档位 | 判定 | 当日上限 | 任务类型 |
|---|---|---:|---|
| **D · 恢复** | 严重缺觉、明显不适、工作被榨空 | 0–30 分钟 | 只保连续性，尽快睡觉 |
| **C · 保底** | 睡眠不足 5.5h，或工作拖尾明显 | 45–60 分钟 | 旧题、错题、选择题、英语维持 |
| **B · 正常** | 睡眠 5.5–6.5h，工作正常收口 | 75–100 分钟 | 一个主任务，限时完成 |
| **A · 高质量** | 睡眠超过 6.5h，或早下班、无加班周末 | 150–240 分钟 | 数学综合题、408 大题、完整训练 |

状态差时控制损失，状态好时集中突破。禁止在严重缺觉日用熬夜补进度。

## 每日任务协议

每天只允许：

- 1 个主任务
- 0–1 个维持任务
- 1 份明确不做清单

主任务必须写清：

```markdown
任务：
完成线：
预计时间：
时间上限：
```

例如：

```markdown
任务：组成原理数据表示真题
完成线：独立完成 8 题，订正并归纳 3 个错因
预计时间：75 分钟
时间上限：90 分钟
今天不做：目录调整、AI 工具研究、额外资料搜索
```

时间到就停止。未完成部分转成下一动作，不在当晚无限延长。

## 睡眠规则

第一阶段目标不是立刻睡满 8 小时，而是把平均睡眠从约 5.3–5.5 小时稳定提高到 6 小时：

- 00:30 停止 Codex、GitHub 构建和高刺激搜索
- 01:00 后不开始新任务
- 01:30 进入洗漱、关机、降刺激流程
- 02:00 前睡着

连续一周稳定后，再将停机和入睡各提前 30 分钟。完整依据见 [睡眠基线](data/sleep/README.md)。

## 工作规则

工作日的首要控制点不是“全天不摸鱼”，而是避免任务在下班前集中爆发。

每天 17:00 左右写：

```markdown
今天必须完成：
做到什么程度即可交付：
可以明天继续：
今天不再新开：
预计下班：
```

下班前 60–90 分钟只做交付和收尾。加班按月预算，不作为默认状态。完整考勤与调度依据见 [工作基线](data/work/README.md)。

## 回家后的切换

默认流程：

```text
到家后 0–40 分钟：吃饭、洗澡、缓冲
第 40 分钟：执行极小进入动作
进入动作：打开目标材料、写日期、做第一题
```

休息必须存在，也必须有结束点。不要等待“完全恢复”再开始。

## 时间管理方法

Horizon 只吸收成熟项目中真正有用的机制：

1. **Timebox**：先设时间上限，不允许任务无限膨胀。
2. **Actual Tracking**：记录预计与实际有效时间，修正容量估计。
3. **WIP Limit**：限制同时进行的任务，避免四科和项目同时争夺晚间窗口。
4. **Leak Review**：每天只标一个主要漏损，每周只修最高频项。

完整研究与取舍见 [时间管理研究](research/time-management/README.md)。

漏损代码：

```text
S：睡眠不足
W：工作拖尾
R：恢复无边界
P：项目替代学习
D：手机 / 网页分心
T：任务过大
U：任务定义不清
```

## 160 天三阶段

### 1. 止漏与稳定

验收条件：

- 平均睡眠达到 6 小时
- 工作日晚间稳定产生 45–90 分钟考试训练
- 每周达到 8–10 小时真实考试训练
- 到家后 60 分钟内进入任务态的比例上升
- 项目建设不再替代训练

### 2. 提高训练密度

第一阶段稳定后，将工作日晚间提高到 75–100 分钟，每周至少形成一个 2.5–4 小时高质量块。重点是做题、订正、重做和限时输出，不是增加课程与资料。

### 3. 考试化

逐步进入专题、真题、大题、套卷和限时书写。此阶段停止系统扩建，Horizon 只做调度、记录和风险预警。

## 系统红线

1. 睡眠不足 5.5 小时，当天禁止补进度。
2. 每天只允许一个主任务。
3. 构建产出不得计入考试训练时长。
4. 工作日先产生考试训练，再允许项目维护。
5. 任何计划必须同时写明“不做什么”。
6. Horizon 每日记录不超过 5 分钟，周维护不超过 60 分钟。
7. 新功能必须解决连续两周出现的真实问题。
8. 当天失败不补偿到凌晨，直接修正下一天。

## 是否有效

Horizon 是否有效，只看四个结果：

1. 睡眠是否逐步增加；
2. 工作是否更容易收口；
3. 每周真实考试训练是否稳定增长；
4. 学习项目是否持续产生可验证的考试产出。

目录、commit 和自动化都只是工具。它们不能替代这四条。
