# Horizon · 四战 160 天控制系统

> Horizon 不是另一份更复杂的计划，也不是用来制造“我已经在努力”的项目。
> 它只负责一件事：在工作、睡眠、休息、学习和多个独立 Git 项目之间，持续做出对上岸最有利的调度。

## 北极星

- **目标院校**：华中师范大学
- **目标方向**：计算机技术 / 软件工程
- **剩余时间**：160 天
- **主目标**：把每天有限的精力，稳定转化为考试产出
- **核心判断**：当前最大问题不是不知道学什么，而是睡眠不足、工作拖尾、恢复无边界、项目建设替代学习，导致精力无法沉淀

Horizon 不再追求“每天四科都推进”，也不再用宏大目标压迫当天。它先堵漏，再增量；先恢复容量，再提高训练强度。

---

## Horizon 管什么

Horizon 是本地总工作区和控制面。它可以包含多个彼此独立的 Git 仓库，但不替这些仓库提交，也不固定只管理 `english2-daily` 和 `408-from-44`。

```text
horizon/
├── .git/                  # Horizon 自己的仓库
├── README.md              # 总规则与当前战略
├── current/               # 当前周、当前日
├── logs/                  # 真实执行记录
├── reviews/               # 周复盘、阶段复盘
├── <任意独立 Git 仓库>/
│   └── .git/
└── <更多独立 Git 仓库>/
    └── .git/
```

本地统一打开一个工作区，减少切窗口和上下文切换；进入不同子目录时，由最近的 `.git` 决定提交到哪个仓库。Horizon 只负责调度，不物理合并其他项目，也不追踪它们的全部内容。

## 系统的五个输入

每天的安排不能只看任务清单，而要先读取真实状态：

1. 昨夜实际睡眠时长
2. 当天工作是否拖尾
3. 实际下班与到家时间
4. 当前精力和启动阻力
5. 当前最需要推进的考试瓶颈

输入错误，计划一定错误。Horizon 不允许根据“理想状态”排当天任务。

## 容量档位

时间不是均质的，精力也不是。每天只进入一个档位。

| 档位 | 判定 | 当日上限 | 适合任务 |
|---|---|---:|---|
| **D · 恢复** | 严重缺觉、明显不适、工作被榨空 | 0–30 分钟 | 只保连续性，尽快睡觉 |
| **C · 保底** | 睡眠不足 5.5h，或工作拖尾明显 | 45–60 分钟 | 旧题、错题、选择题、英语维持 |
| **B · 正常** | 睡眠约 5.5–6.5h，正常 20 点下班 | 75–100 分钟 | 一个主任务，限时完成 |
| **A · 高质量** | 睡眠超过 6.5h，或早下班 / 无加班周末 | 150–240 分钟 | 数学综合题、408 大题、完整训练 |

规则不是“状态差也尽量多做”，而是：**状态差时控制损失，状态好时集中突破。**

## 每日只允许一个主任务

每天只定义一个真正的主任务。其他内容只能是维持项。

主任务必须写清三件事：

```markdown
主任务：组成原理数据表示真题
完成线：独立完成 8 题，订正并归纳 3 个错因
时间上限：90 分钟
```

必须同时写：

```markdown
今天明确不做：Horizon 结构调整、AI 工具研究、额外资料搜索
```

“今天不做什么”不是附注，而是控制系统的一部分。未被选择的任务是主动舍弃，不是心理债务。

## 睡眠：第一优先级的容量工程

Apple 健康数据表明，近 100 天夜间主要睡眠长期约 5.5 小时，最近 30 天几乎每天不足 6 小时。当前问题不是偶尔熬夜，而是稳定地以凌晨 3 点左右入睡、早上 8 点左右起床。

第一阶段不要求立刻睡满 7–8 小时，只做渐进修复：

### 阶段 S1：先稳定到 6 小时

- 00:30 后不再使用 Codex、Claude Code 或做项目建设
- 01:00 后不开始新任务
- 01:30 进入洗漱、关机、降刺激流程
- 02:00 前睡着

连续 7 天大多数夜晚达到 6 小时后，再把停机和入睡时间各提前 30 分钟。

睡眠不是学习结束后的剩余时间，而是第二天数学、408、工作收口和身体恢复的预算。严重缺觉日禁止补进度。

## 工作：把疲劳留在公司，不带回家

当前工作日通常约 8:50 上班、20:05 下班，电瓶车十几分钟到家。真正消耗晚间的并不只是下班晚，而是白天摸鱼后任务拖到最后，导致下班前被工作集中榨干。

工作日只执行一条关键规则：

> 下班前 60–90 分钟停止无目的摸鱼，只做当天必须收口的工作。

每天 17 点左右完成一次工作切片：

```markdown
今天必须完成：
可以明天继续：
今天不再新开：
```

目标不是在公司更拼命，而是让下班时形成明确的终点，降低回家后的恢复成本。

加班属于财务任务，不是默认状态。每月应先确定需要的收入和对应加班额度，超过额度后的加班必须重新比较：多挣的钱，是否值得交换睡眠和训练容量。

## 休息：必须恢复，也必须结束

回家后需要恢复，但恢复不能无限延长。

建议默认流程：

```text
到家后 0–40 分钟：吃饭、洗澡、坐下缓冲
第 40 分钟：执行一个极小的进入动作
进入动作：打开目标文件 / 写日期 / 做第一题
```

不要等待“完全恢复好”再开始。很多时候状态是在第一题之后恢复，而不是在刷手机中恢复。

休息的完成标准不是“我感觉够了”，而是“到了预设时间，我完成了从休息态到任务态的切换”。

## 项目建设的边界

GitHub、Codex、Claude Code 可以提高构建产出，但构建产出不能冒充学习产出。

允许的项目使用：

- 批改已经完成的英语训练
- 整理已经做过的错题和错因
- 生成下一次明确训练任务
- 自动化已经稳定运行两周的重复流程

禁止优先发生的事情：

- 先改架构，再开始学习
- 反复设计目录、协议和世界观
- 用 commit 数量代替考试产出
- 在工作日晚间扩展 Horizon 功能

硬规则：

> 当周没有真实考试训练产出时，不允许新增 Horizon 功能。

Horizon 每周维护时间默认不超过 60 分钟，日常记录不超过 5 分钟。

## 时间管理方法：借机制，不复制工具

Horizon 吸收成熟时间管理项目中的三个机制：

### 1. Timeboxing：先承诺时间上限

不是列出很多待办，而是给唯一主任务一个固定时间盒。时间到就停止，避免一道题扩展成两小时搜索、讨论和结构调整。

### 2. Actual Tracking：记录真实耗时

每天记录计划分钟和实际有效分钟。估算误差比“完成 / 未完成”更有价值。

```markdown
计划：90 分钟
实际有效：62 分钟
误差原因：启动晚 20 分钟，AI 扩展 8 分钟
```

### 3. Leak Diagnosis：每周只修一个最大漏洞

漏损代码：

```text
W：工作拖尾
S：睡眠不足
R：恢复无边界
P：项目替代学习
T：任务过大
D：手机 / 网页分心
U：任务定义不清
```

每天最多标记一个主要漏损。周复盘时统计最高频项，下周只修这一项。

## 160 天推进原则

### 第一阶段：止漏与稳定

目标：恢复基本容量，不追求四科同时猛推。

验收：

- 平均睡眠达到 6 小时
- 工作日晚间可稳定产生 45–90 分钟考试训练
- 每周至少 8–10 小时真实考试训练
- 项目建设每周不超过 2 小时
- 到家后 60 分钟内进入任务态的比例持续上升

### 第二阶段：提高训练密度

在第一阶段稳定后，把工作日晚间提高到 75–100 分钟，并确保每周至少出现一个 2.5–4 小时高质量块。

重点不是增加课程和资料，而是提高单位时间内的考试产出：做题、订正、重做、限时输出。

### 第三阶段：考试化

逐步从章节训练转向专题、真题、大题、套卷和限时书写。Horizon 此时停止结构开发，只做调度和记录。

## 系统红线

1. 睡眠不足 5.5 小时，当天禁止补进度。
2. 每天只允许一个主任务。
3. 项目构建不得计入考试训练时长。
4. 工作日先完成考试训练，再允许项目维护。
5. 任何计划必须同时写明“不做什么”。
6. Horizon 维护时间不得高于真实训练时间。
7. 连续两周没有执行价值的功能，不开发。
8. 当天失败不补偿到凌晨，直接修正下一天。

---

## 最终判断标准

Horizon 是否有效，不看目录是否漂亮，不看 commit 是否频繁，只看四个结果：

1. 睡眠是否逐步增加
2. 工作是否更容易收口
3. 每周真实考试训练是否稳定增长
4. 学习项目是否持续产生可验证的考试产出

只要这四条在改善，系统就在把四战从“想上岸”推向“有能力上岸”。
